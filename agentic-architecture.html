Sub-Agent Delegation: How It Works End-to-End

What is a sub-agent?

A sub-agent is a named specialist you pre-define in the config file. Each one has its own AI model, personality, and limited set of tools. Think of it like hiring a freelancer: you describe who they are and what they can do, but they only start working when the main AI decides it needs help.

Sub-agents do NOT run on their own. They only exist when the parent AI explicitly calls the delegate tool — just like any other tool (shell, file, memory). No separate process, no separate server.

Step 1: You define sub-agents in config
[agents.researcher]
provider = "anthropic"
model = "claude-4"
system_prompt = "You are a research specialist"
agentic = true
allowed_tools = ["web_search", "read_file"]
max_depth = 3
max_iterations = 10

provider + model — which AI to use (can be completely different from the parent)

system_prompt — the sub-agent's personality and instructions

agentic = true — can use tools in a loop (not just answer once)

allowed_tools — only these tools are available (filtered from parent's tools)

max_depth — how many levels of sub→sub delegation are allowed

max_iterations — safety limit on tool loop cycles

Step 2: The parent AI decides to delegate (not you)

When you define sub-agents in config, the system automatically adds a delegate tool to the parent AI's tool list. The parent AI sees it like any other tool and chooses to use it when a task fits a specialist.

What the parent AI returns:
tool_call: delegate(agent="researcher", prompt="Find competitor pricing for Q2")
→ The Thinking Loop sees this tool call and runs it, just like it would run shell() or memory_save(). The AI model decides; the system executes.

Identity & Personality: What Gets Sent to the AI Every Time

Before the AI sees your message (Step 4 above), the system assembles a system prompt that tells the AI who it is, how to behave, what it knows, and what tools it has. This prompt is built from files in your workspace:

SOUL.md
Core personality, values, and behavioral rules
USER.md
Info about you (preferences, context the AI should know)
IDENTITY.md
Name, role, description
AGENTS.md
Project conventions and rules
TOOLS.md
Custom tool instructions
How it works
1 On every message, the system reads these files from your workspace folder
2 Each file is trimmed to 20,000 characters max (6,000 in compact mode)
3 They're combined into one system prompt: safety rules + identity + skills + memory + tool list
4 This system prompt is the first message in every AI conversation
Key behaviors
File exists and has content?
Included in the system prompt, in full (up to the char limit).
File is empty?
Silently skipped — nothing is added for that file.
File doesn't exist?
A note is added: "[File not found: SOUL.md]" so the AI knows it's missing.
Alternative format: Instead of markdown files, you can use a single AIEOS JSON file that defines identity, personality, tone, and context in a structured format. Set identity.format = "aieos" in your config.
What the AI actually receives (system prompt assembly order):
Safety Rules → SOUL.md + USER.md + IDENTITY.md → Available Tools → Skills → Recalled Memories → AGENTS.md + TOOLS.md
This entire prompt is sent as the first message every time the AI is called. Your actual question and conversation history follow after it.

Session Management: How Conversations Are Tracked

Every conversation has a session — the system keeps track of what was said so the AI has context for follow-up messages. Here's how it works end to end:

What defines a "session"?

Each conversation is identified by a unique key made of:

{channel} + {chat_id} + {thread} + {sender}
Example: You message via Telegram in group chat #marketing, thread "budget":
telegram_marketing_budget_user42
This means:
  • Different chats have separate histories
  • Different threads within a chat are separate
  • Different users in the same chat get their own context
Where are sessions stored?
While running (in memory):
  • Fast in-memory cache holding up to 1,000 sessions
  • Each session keeps the last 50 messages
  • Oldest sessions are evicted when the cache is full
On disk (survives restarts):
  • Each session is saved as a file in workspace/sessions/
  • Messages are appended line-by-line (append-only log)
  • On startup, the most recent 1,000 sessions are reloaded
  • Enabled by default — set session_persistence = false to disable
Context Window: How the AI Sees Your Conversation

AI models have a limited "memory" per call (the context window, typically 32K–128K tokens). The system manages this carefully to fit as much relevant context as possible:

1. Token Budget
Before each AI call, the system estimates how many tokens the full conversation uses (~4 characters = 1 token). Default budget: 32,000 tokens.
2. Smart Trimming
If over budget: old action results are compressed first (keep first 2/3 + last 1/3). Then older messages are dropped, keeping the most recent 4 messages intact.
3. Context Compression
For long conversations, the AI itself summarizes older history into a condensed recap, keeping the thread of the conversation without the full verbatim text.
4. Emergency Recovery
If the AI model rejects a request because it's too long, the system automatically trims old action details + drops the oldest 1/3 of messages, then retries.
5. Tool Context Cleanup
Only the last 2 turns keep full action details. Older turns have their action context stripped to save space while keeping the conversation flow.
What the AI sees on each call (in order):
System Prompt contains SOUL.md USER.md tools memories
then Conversation History up to 50 messages, trimmed to fit token budget
then Your Latest Message the new question or follow-up
Config fields that control this: max_history_messages = 50, max_context_tokens = 32000, keep_tool_context_turns = 2, compact_context = false

The Thinking Loop — How the Agent Reasons +

This is the brain of the system. When a message arrives, it goes through a repeating cycle:

  1. Build context — The system combines your message with the agent's personality, relevant memories, skill instructions, and language settings into one package for the AI.
  2. Ask the AI — Sends everything to the chosen AI model. As the AI writes its response, partial updates are streamed back so you see the reply forming in real time.
  3. Take action — If the AI decides it needs to do something (search the web, save a file, check the weather, etc.), the system runs a safety check, optionally asks for your approval, then executes the action.
  4. Repeat — If the AI needs more steps, the cycle continues. Once the AI has a final answer, it's sent back to you.

Supporting features: message classification (understands what you're asking), memory retrieval (pulls in relevant past context), budget tracking (watches spending), reasoning traces (shows the AI's thought process), and conversation history management.

Action Parser — Understanding What the AI Wants to Do +

When the AI decides it needs to take an action (like searching the web or saving a file), it expresses that in its response. The Action Parser reads the AI's output and figures out exactly what action to run. It works in two ways:

  • Built-in support — Modern AI models (like GPT-4 and Claude) can directly specify actions in a structured format. No guesswork needed.
  • Text parsing — For simpler AI models that don't have built-in action support, the system reads specially formatted text in the AI's response to figure out what it wants to do.

After parsing, the action request and its results are formatted and fed back into the conversation so the AI can use them in its next response.

Skills & App Connectors — Extending What the Agent Knows +

Skills are add-on capability packs you can install — like apps for your phone. Each skill teaches the agent something new (e.g., "how to deploy a website" or "how to manage a database"). Skills can be browsed, installed, safety-checked, and tested before use.

Runtime, Settings, Document Search, Multi-Agent +
Runtime

Controls where the agent runs its commands. Two options: locally on your machine, or inside a Docker container (an isolated virtual environment) for added safety.

Settings

One configuration file controls the entire system: which AI to use, which messaging platforms to connect, security rules, memory storage, scheduled tasks, monitoring, integrations, and more. Workspaces keep different projects isolated from each other.

Document Search

The agent can search through reference documents (like PDF manuals or datasheets) to find specific information. It breaks large documents into searchable chunks and retrieves the most relevant sections when you ask a question.

Multi-Agent & Delegation

The AI can hand off tasks to sub-agents — each with its own model, personality, and allowed tools. For example, a main GPT-4 agent can delegate research to a Claude-based "researcher" sub-agent that only has web search access. Sub-agents run inside the same system (no separate process) and return their results as normal tool output.

Background mode: long-running tasks run asynchronously; the parent gets a task ID and can check progress or cancel later. Swarm mode: orchestrate multiple agents at once — sequentially, in parallel, or with a router that picks the best specialist for each task.

Nodes are a separate feature — they connect multiple independent agent instances across machines via the gateway for distributed work.

Security — Keeping Everything Safe +
Autonomy Levels

You decide how much freedom the agent has. Three modes: Read-Only (can only look, never change anything), Supervised (asks you before doing anything risky), and Full (acts independently). The system also classifies how risky each action is and limits how often any single user can trigger actions.

Sandbox (Isolation)

When the agent runs commands on your computer, it can do so inside an isolated "sandbox" that prevents it from touching anything outside its allowed workspace. The system automatically picks the best isolation method for your operating system:

DockerFirejailBubblewrapLandlock (Linux)Seatbelt (macOS)
Human Approval

For high-risk actions, the agent pauses and asks for your confirmation before proceeding. You can approve once, deny, or say "always allow this" so it won't ask again for that action.

Trust Score & Action Proof

The system tracks how well the agent is performing over time. If it makes mistakes, its trust score drops and it may lose autonomy. For critical actions, cryptographic signing creates a tamper-proof record of exactly what was authorized and by whom.

Additional protections: prompt injection detection (blocks manipulation attempts), access control policies, hardware-key authentication, emergency stop (halt everything instantly), secrets management (safe storage for passwords/keys), and audit logging (full record of every action).

Scheduled Tasks — Making the Agent Work on Autopilot +

All automation features feed work back through the same thinking cycle, so the agent handles scheduled tasks exactly like user messages:

Schedules

Run tasks on a timer: every hour, daily at 9am, once next Tuesday, or after a 5-minute delay. Scheduled jobs can be simple commands or full AI conversations with specific instructions.

Playbooks

Multi-step procedures the agent follows, like a recipe. Example: "When a new deploy happens, run tests, check logs, and post a summary to Slack." The agent can list available playbooks and run them on demand.

Auto-Rules

"When X happens, do Y" rules. For example: "When someone mentions 'urgent', send a notification" or "When a new file is saved, run the linter." Rules can trigger messages, playbooks, commands, or new scheduled tasks.

Recurring Jobs

Background tasks that run periodically with their own context. Each run is logged so you can see what happened and when.

Event Hooks

Custom logic that runs at key moments in the agent's lifecycle: when the server starts, when a session opens, before/after the AI responds, before/after an action runs, or when a message is sent. Useful for logging, filtering, or modifying behavior.

Heartbeat & Background Mode

The agent can run as a background service on your computer (like a system service). It periodically checks in with a "heartbeat" to confirm it's still running, and can be managed with simple start/stop/restart commands.
