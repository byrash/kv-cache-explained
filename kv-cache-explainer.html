KV Cache

The optimization that makes real-time LLM inference possible

1. The Problem: Redundant Recomputation

First — How Does an LLM Actually Generate Text?

An LLM does not generate a whole sentence at once. It works like someone typing one letter at a time, where each letter depends on everything typed before. Specifically:

Given a sequence so far, the model predicts only the next token. Then it appends that token to the sequence and predicts again. And again. And again. This is called autoregressive generation.

Let's trace through generating "The cat sat on the mat" step by step. Suppose the user typed nothing — the model starts with just a seed:

Notice: at Step 3, the model feeds ["The", "cat", "sat"] — all three — just to predict one word "on".

Why Does It Need All Previous Tokens?

Because context matters. The next word depends on the meaning built from all prior words:

"The cat sat on the ___" → model predicts "mat" (because cats sit on things)
"The bird sat on the ___" → model might predict "branch" (different subject = different prediction)
"The CEO sat on the ___" → model might predict "board" (different context entirely)
The model must "re-read" the whole sentence each time to understand context. Changing even one early word changes everything downstream.

The Redundancy Problem

Here's where it gets wasteful. The model doesn't just "read" tokens — it does heavy math on each one. For every token, it computes internal vectors (we'll call them K and V for now — explained in the next section). Watch the waste grow:

Look at "The": its internal vectors are computed at Step 1, then thrown away, then recomputed identically at Step 2, thrown away again, recomputed at Step 3... Every orange block is wasted work.

Why is this so bad? For a sequence of n tokens, total work = 1 + 2 + 3 + ... + n = n(n+1)/2. That's O(n²).

Real numbers: Generating a 1,000-token response = ~500,000 token-level computations. A 10,000-token response = ~50,000,000. It grows quadratically — every extra token makes ALL previous tokens get reprocessed.

The Key Realization

Here's the critical observation that makes KV Cache possible:

The internal vectors (K and V) computed for "The" at Step 1 are exactly identical to the ones computed for "The" at Step 2, 3, 4, ...

Why? Because K and V for a token depend only on: (1) the token itself, and (2) its position in the sequence. Neither of these changes between steps. Future tokens don't affect past tokens' K,V values — this is enforced by the causal mask in the attention mechanism.

So why recompute them? Just save them the first time! That's KV Cache.

WQ, WK, WV are learned during training and fixed during inference. Same matrices used for every token.
Attention(Q, K, V) = softmax( Q · KT / √dk ) · V
Q·KT = "how relevant is each token?"  |  softmax = "normalize to weights"  |  ·V = "blend the information"

Why K and V Never Change — The Crucial Point

Kcat = embedding("cat") × WK
Vcat = embedding("cat") × WV

Both the embedding of "cat" and the weight matrices WK, WV are fixed during inference. They don't change when you add "sat" or "on" later. So Kcat computed at Step 1 is byte-for-byte identical to Kcat at Step 5.

This is the entire foundation of KV Cache: since K and V for past tokens are deterministic and immutable, compute them once, store them, and reuse them forever.

Why NOT Cache Q?

Think about it with our example:

  • At Step 3, when we predict "on", the Query for position 4 asks: "What follows 'The cat sat'?"
  • At Step 4, when we predict "the", the Query for position 5 asks: "What follows 'The cat sat on'?"
  • These are different questions from different positions. We never ask the same Q twice.
  • But Kcat ("I'm a noun, the subject") is needed at Step 2, 3, 4, 5, ... — every future token needs to know about "cat".
  • So: Q is use-once, K is use-forever, V is use-forever. Cache the ones used forever.

3. KV Cache: The Solution

The idea is beautifully simple: store all previously computed Key and Value vectors in GPU memory, so each generation step only needs to:

  1. Compute Q, K, V for only the new token
  2. Append the new K,V to the cache
  3. Read all previous K,V from the cache
  4. Run attention: new Q against all cached K's → weight all cached V's

Step-by-Step Walkthrough

Result: Each step does exactly 1 new computation + a cache read. Total for n tokens: O(n) instead of O(n²). In practice this means 10-100x faster inference.

Why Only K and V? Why Not Q?

The Query vector is only ever used at the step it's created — it asks "what's relevant to ME?" and then it's done. We never need a past token's Q again. But every past token's K (its identity label) and V (its information content) must be available for ALL future tokens to attend to. That's exactly what the cache stores.

Key Insight: Most of the wall-clock time is spent in the Decode phase. Prefill is fast (parallel), but Decode is limited by memory bandwidth — reading the ever-growing KV cache from GPU VRAM. This is why KV cache optimization matters so much.

Why Decode is Memory-Bound

During decode, the GPU is doing very little math (just 1 token's worth) but reading a lot of data (the entire KV cache). Modern GPUs have far more compute power than memory bandwidth, so the GPU sits mostly idle, waiting for data. This is why techniques like quantized KV cache (reducing bytes per value) directly speed up decode.

6. KV Cache Memory: The Elephant in the Room

The KV cache lives in GPU memory (VRAM) and grows linearly with sequence length, across every layer and every attention head.

KV Cache Size = 2 × num_layers × num_kv_heads × head_dim × seq_len × bytes_per_param
2 = one for K, one for V  |  bytes = 2 for FP16, 1 for INT8, 0.5 for INT4

B. PagedAttention (vLLM)

Inspired by OS virtual memory. Instead of allocating contiguous GPU memory per sequence (wasteful due to padding), allocate small fixed-size pages:

Traditional: Pre-allocate max_seq_len for each request → massive waste
PagedAttention: Allocate in small pages, mapped via page table → near-zero waste, pages can be shared across beam search candidates

C. Quantized KV Cache

Reduce precision of cached K,V values. K,V are less sensitive to quantization than model weights:

D. Sliding Window Attention

Only attend to the last W tokens instead of the full history. Cache holds at most W entries:

Used by Mistral (W=4096). Some layers use full attention while others use windowed — hybrid approach preserves long-range understanding while saving memory.

E. KV Cache Eviction (H2O / StreamingLLM)

For truly infinite-length generation, intelligently evict old entries:

  • H2O (Heavy Hitter Oracle): Keep tokens that consistently receive high attention scores ("heavy hitters"), evict low-attention ones
  • StreamingLLM: Keep "attention sinks" (first few tokens that anchor the attention distribution) + a sliding window of recent tokens. Enables infinite-length generation with fixed memory.

In one sentence: KV Cache is memoization for attention — it stores the Key and Value vectors of all past tokens so they're never recomputed, turning quadratic inference into linear, and making real-time chat with LLMs possible.

The Fundamental Trade-off

Speed vs Memory. KV Cache trades GPU memory for dramatically faster generation. Every major optimization (GQA, quantization, paging, eviction) is about getting the same speed benefit while using less memory.

Why It Matters

  • Without KV cache, ChatGPT-style real-time streaming would be impossible
  • It's the reason long-context models (128K+ tokens) need massive GPU clusters
  • KV cache management (vLLM's PagedAttention) is what makes serving thousands of concurrent users feasible
  • The MHA→GQA evolution was primarily motivated by KV cache reduction
  • Quantized KV cache directly speeds up decode by reducing memory bandwidth demands
