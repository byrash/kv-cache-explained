Part I — How Claude Code Talks to Models

MEM Memory & Preferences

All preferences (CLAUDE.md, auto-memory, team memory) are sent on every API call. The model has no persistent local state.

Is CLAUDE.md cached locally or sent to the model every time?
Both. Files are read from disk once per session (memoized), then the same content is included in every API call as messages[0]. Anthropic's prompt caching (cache_control: ephemeral) avoids re-processing the stable prefix server-side — but the bytes still travel over the wire every time.

The 4-Tier Hierarchy

Loaded in order — later files have higher priority (model pays more attention to content near the end).

1. Managed
/etc/claude-code/CLAUDE.md — Set by IT admins. Lowest priority.
2. User
~/.claude/CLAUDE.md — Personal global preferences for all projects.
3. Project
CLAUDE.md, .claude/CLAUDE.md, .claude/rules/*.md from repo root → CWD. Shared with team.
4. Local
CLAUDE.local.md — Private, gitignored. Highest priority.

Plus: AutoMem (~/.claude/projects/<repo>/memory/) and TeamMem (server-synced shared memory).

POST to platform.claude.com/v1/oauth/token. Store: macOS keychain (hex-encoded, 30s read cache) or ~/.claude/.credentials.json (Linux).

Token Refresh

Pre-emptive Refresh

Tokens checked before every API call. Considered expired 5 min before actual expiresAt (clock drift buffer).

Cross-Process Locking

File-based lock on ~/.claude/ coordinates refresh across concurrent instances. 5 retries, 1-2s randomized backoff.

401 Recovery

Deduplicates concurrent 401s. Checks if keychain has a fresher token (another instance refreshed). Force-refreshes if same token.

Part II — Security & Isolation

SBX Sandboxing

Two-layer OS-level isolation: filesystem via Seatbelt (macOS) / bubblewrap (Linux), network via proxy filtering. Kernel-enforced, not bypassable.

macOS — Seatbelt

Default-deny profile generated
(version 1)(deny default) — blocks all operations. Then specific allows added.
Filesystem: read broadly, write narrowly
(allow file-read* (subpath "/")) then (allow file-write* (subpath "/project")). Later rules override. Move blocked via (deny file-write-unlink).
Network: localhost-only proxy
Only (allow network-outbound (remote tcp "localhost:PROXY_PORT")). All other network blocked.

Linux — Bubblewrap

Read-only root + selective bind mounts
--ro-bind / / then --bind /project /project for writable dirs. Deny-within-allow via --ro-bind /dev/null <path>.
Network + PID namespace isolation
--unshare-net (empty namespace, zero interfaces) + --unshare-pid. Network restored via socat Unix socket bridges to host proxy.
seccomp-bpf blocks Unix sockets
Two-stage: socat bridges connect first (allowed), then apply-seccomp installs BPF filter blocking socket(AF_UNIX) before exec'ing user command.

Network Proxy

HTTP CONNECT Proxy

Handles HTTPS via CONNECT tunnels + plain HTTP forwarding. Calls filterNetworkRequest() per connection. Returns 403 for blocked domains.

SOCKS5 Proxy

For SSH/git traffic. Same allowlist. On Linux, socat bridges connect sandbox's isolated netns to host proxies via Unix sockets.

Domain Filtering

Reject malformed hosts. Canonicalize IPs (prevent inet_aton bypass). Check deny → allow → ask user. Wildcard *.example.com supported. Rejects wildcard on IP literals.

Part III — Applying to Copilot CLI

GH Copilot CLI Implementation

GitHub's new Copilot CLI (npm install -g @github/copilot) is a full agentic CLI — file editing, command execution, /plan, /fleet, MCP, session persistence. But it has no sandboxing or hooks.

Realistic Options

A. AGENTS.md + Org Policy

Push instructions via GitHub admin. Zero custom code. Not a security boundary — model can be tricked.

B. Wrap Whole Process with SRT

srt npx @github/copilot — sandboxes everything. May need domain allowlisting for Copilot's API calls.

C. Surgical: Sandbox Only Spawned Commands

Set SHELL=/path/to/sandboxed-bash. Copilot runs free (API calls work), but every command it executes is sandboxed. Best practical option.

D. Build Custom CLI (GitHub Models API)

Full control. Your own agentic loop + tools + sandbox + hooks. ~2 weeks MVP. Most effort, most power.

Defense in Depth (3 Layers)

AGENTS.md
Policy layer — guides model behavior. Not a security boundary. Model can be prompt-injected.
Permission Rules
Software gate — allow/deny/ask patterns. Reduces prompt fatigue. Defense layer, not enforcement.
OS Sandbox (SRT)
Hard enforcement — kernel blocks syscalls. Filesystem: Seatbelt/bwrap. Network: proxy filtering. True security boundary.

Programmatic Usage

SRT as a Library
import { SandboxManager } from '@anthropic-ai/sandbox-runtime'

// 1. Initialize — starts proxy, loads config
await SandboxManager.initialize({
  network: { allowedDomains: ['github.com', '*.github.com'] },
  filesystem: { allowWrite: ['.'], denyRead: ['~/.ssh'] }
})

// 2. Wrap command — returns sandbox-exec (macOS) or bwrap (Linux) string
const cmd = await SandboxManager.wrapWithSandbox('npm test')

// 3. Execute — child inherits all restrictions
const child = spawn(cmd, { shell: true, stdio: 'inherit' })

// 4. Cleanup mount-point artifacts (Linux)
child.on('exit', () => SandboxManager.cleanupAfterCommand())
Can SRT sandbox any process, not just Claude Code?
Yes. It's model-agnostic. Sandbox anything: srt <any-command>. Works with Copilot CLI, your own CLI, or non-AI processes.

WSL Windows / WSL2 Sandboxing

Claude Code has no native Windows sandbox — it uses WSL2 + bubblewrap instead. The same approach works for sandboxing Copilot CLI on Windows fleets.

Why WSL2 Works

Real Linux Kernel

WSL2 runs a full Hyper-V Linux kernel. CLONE_NEWNS, CLONE_NEWNET, CLONE_NEWPID all work — bubblewrap functions identically to native Linux.

WSL1 Is a Hard No

WSL1 translates syscalls — no real namespaces. Bubblewrap cannot run. Must enforce WSL2 via wsl --set-default-version 2 across fleet.

SRT Auto-Detects

@anthropic-ai/sandbox-runtime sees Linux and uses bubblewrap + socat automatically. No WSL-specific code needed.

Network Stack

WSL2 has its own virtual network adapter. Proxy-based domain filtering works identically — all egress passes through the SRT proxy.

Setup Steps

Ensure WSL2 is installed
MDM can push via wsl --install (supported since Win10 2004). Force version 2: wsl --set-default-version 2.
Install sandbox dependencies inside WSL2
sudo apt-get install bubblewrap socat — same as native Linux.
Install Copilot CLI inside WSL2
Node.js + npm live in WSL2, not the Windows side: npm install -g @github/copilot.
Wrap with SRT
npx @anthropic-ai/sandbox-runtime gh copilot — sandboxes everything. Same command as native Linux.
Access project files
Windows repos accessible via /mnt/c/Users/.... For better performance, keep repos inside WSL2 filesystem (~/projects/).

OS Detection Wrapper

Cross-Platform Launcher Script
#!/usr/bin/env bash
if [[ "$(uname -r)" == *microsoft* ]]; then
  # Already inside WSL2 — use SRT directly
  exec npx @anthropic-ai/sandbox-runtime "$@"
elif [[ "$(uname)" == "Darwin" ]]; then
  # macOS — SRT uses Seatbelt
  exec npx @anthropic-ai/sandbox-runtime "$@"
elif command -v wsl.exe &>/dev/null; then
  # Windows-side — delegate into WSL2
  exec wsl -- npx @anthropic-ai/sandbox-runtime "$@"
else
  # Native Linux — SRT uses bubblewrap
  exec npx @anthropic-ai/sandbox-runtime "$@"
fi
